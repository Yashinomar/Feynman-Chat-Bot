1. PROJECT OVERVIEW AND WHAT WE HAVE DONE SO FAR
------------------------------------------------
The TeachBack AI (formerly Feynman-Chat-Bot) project is an innovative educational platform built fundamentally around the Feynman Technique: the concept that you learn best by teaching someone else. Over the course of the development cycle, we have transitioned from a simple concept to a fully realized, multi-modal web application utilizing state-of-the-art Next.js App Router mechanics and Google's Gemini LLMs.

Our primary goal was to create an interactive learning environment where a user can enter a topic, and an AI-driven "student" persona forces them to explain that topic in clear, simple terms. The AI acts as a naive learner, reacting based on predefined parameters like age and skepticism, scoring the user's explanation, and pointing out gaps in their knowledge.

What we have accomplished so far:
1. Designed and implemented a robust Next.js frontend with stunning, glassmorphism-inspired UI components.
2. Built a dynamic API routing layer capable of handling conversational state, Retrieval-Augmented Generation (RAG) for document ingestion, fact-checking against Wikipedia, and dynamic quiz generation.
3. Implemented a local transcription API utilizing Xenova/Whisper for entirely offline, browser-side to server-side voice processing, dropping dependency on expensive third-party speech-to-text APIs.
4. Created a deeply customizable AI persona configurator, allowing users to teach a "Kid", "High Schooler", or "Professional", with behaviors ranging from "Curious" to "Distracted".
5. Built out multiplayer "Co-Op" modes including Classic, Survival, and Rush Timer modes, supported by basic signaling APIs.
6. Engineered a RAG-based document ingestion pipeline allowing users to upload PDF textbooks. The application parses these documents using pdfjs-dist, slices them into semantic chunks, generates vector embeddings using Google's embedding models, and caches them in memory. The AI learner then compares the user's explanation against their own actual textbook to fact-check them!
7. Built an automated Wikipedia fact-checking fallback that runs in the background.
8. Integrated an automated 10-question multiple-choice quiz generator to test mastery.
9. Styled everything using vanilla CSS Modules, maintaining a heavily stylized, glowing, reactive dark-mode aesthetic with interactive hover states.

The sheer progression of the project represents a massive leap in educational technology capabilities. We have moved from simple textbook reading to highly engaging, active recall generation.

2. IN-DEPTH FEATURE BREAKDOWN AND IMPLEMENTATION DETAILS
--------------------------------------------------------
The feature set of TeachBack AI is highly expansive. Here is exactly what the features are and how they were meticulously constructed:

A) Dashboard and Session Management (page.tsx)
The entry point of the application is a deeply interactive dashboard. It uses React's 'useState' and 'useEffect' heavily to read from 'localStorage' under the key 'feynman_sessions'. We chose local storage for the prototype/alpha phase to guarantee instant load times and zero-latency session retrieval without database overhead. The dashboard displays aggregate statistics like "Topics Learned", "Avg. Mastery", and "Concepts Explained" by reducing the values stored in these session objects.
The dashboard presents the user with four core options:
- Solo Study
- Co-Op Multiplayer
- RAG Notes Study
- Knowledge Quiz

B) The Persona Configurator Modal
Before launching a session, the user is intercepted by a modal that lets them define the LLM's system prompt dynamically. They can choose:
- Student Level: Kid (Lower Primary), High School, Professional.
- Student Behavior: Curious (asks follow-ups), Skeptical (pushes back on shaky logic), Enthusiastic (high energy), Distracted (needs summaries).
Behind the scenes, these selections map to specific string injections in our '/api/chat' server-side route, fundamentally altering how the Gemini model generates its replies.

C) The AI Chat Pipeline (/api/chat/route.ts)
This is the core engine. When the frontend sends a chat payload, we evaluate:
- The conversation history (sliding window of last 10 messages to save tokens).
- The game mode rules (e.g., Rush mode forces the AI to reply in short sentences).
- RAG Context: If the session was started from a PDF upload, we take the user's latest message, generate a vector embedding using 'gemini-embedding-001', and run a cosine similarity calculation against the cached textbook vectors. We grab the top 3 most relevant passages from the user's own PDF and inject them invisibly into the LLM prompt. This forces the AI to say things like "Wait, but your notes said X, why are you saying Y?".

D) PDF Document Parsing and Embedding (/api/rag/route.ts)
When a user uploads a PDF in the dashboard, the browser uses 'pdfjs-dist' to extract the text, appending all pages into a massive string. This string is sent to our RAG route.
The backend takes this massive string and splits it intelligently by sentences (using regex /(?<=.)\s+/ to avoid clipping acronyms). It bundles sentences into "semantic passages" capping at 1200 characters to maximize the signal-to-noise ratio for the embedding model.
Each passage is sent to Google's embedding model. The resulting 3072-dimensional arrays (Float32) are stored in a volatile Node.js server memory cache object ('serverMem.courseNoteMaps') indexed by the session ID.

E) Background Wikipedia Fact-Checking (/api/factcheck/route.ts)
We implemented a secondary validation layer. Whenever a user asserts an explanation, a background fetch is sent to our fact-check route.
This route pings the raw Wikipedia API, searching for the topic. It grabs the first 5 sentences of the official Wikipedia extract. It then feeds both the Wikipedia truth baseline AND the user's explanation into a secondary Gemini-2.5-flash inference call with a temperature of 0.1 (making it highly deterministic). The LLM is instructed to return raw JSON indicating 'isAccurate'. If false, it provides a 1-sentence correction. This prevents users from teaching the AI completely hallucinated physics.

F) Local Voice-to-Text Integration (/api/transcribe/route.ts)
To support voice input without incurring massive AWS or OpenAI Whisper API costs, we implemented Xenova/transformers.js directly on the server.
The user speaks into the microphone. The browser captures the audio, downsamples it, and sends a Float32Array to the backend. The Next.js API route loads 'Xenova/whisper-tiny.en' (a quantized Whisper model weighing only ~40MB) entirely into local memory. It runs the inference on the server CPU, dramatically speeding up transcription and ensuring complete privacy. The resulting text is then pasted into the chat box automatically.

G) Quiz Generator (/api/quiz/route.ts)
A standalone feature generating 10-question multiple-choice quizzes dynamically on any topic requested by the user. Structured JSON schema enforcement ensures the UI can safely map the 'question', 'options' array, and 'answer' directly into a state-driven game loop.

3. THE BACKEND ARCHITECTURE AND TECHNOLOGIES
--------------------------------------------
While traditional applications might use Express.js or Django, TeachBack AI relies exclusively on the Next.js 14/15 App Router Edge/Node API Routes.

The Backend Stack Includes:
- Next.js API Routes (Node.js runtime environment)
- @google/genai SDK for accessing the Gemini family of models (Gemini 2.5 Flash for rapid conversation and JSON parsing, Gemini Embedding 001 for dense vector representations).
- @xenova/transformers for running HuggingFace models (Whisper) locally in JavaScript.
- Node.js global context (global object) for high-speed, volatile RAM caching of vector embeddings and session maps during the MVP phase.

Data Flow Architecture:
1. Client actions trigger fetch() requests to /api/* endpoints.
2. The endpoints deserialize JSON payloads, perform backend validation, and interact with external APIs (Wikipedia, Google GenAI).
3. Complex algorithms (Cosine Similarity for vector matching) are executed synchronously in the route handlers.
4. Clean JSON responses are piped back to the client.

State Management is currently handled by combining React Context / Local State with 'localStorage'. While scalable for an MVP, future backend plans involve integrating Supabase or PostgreSQL with pgvector to persist vector embeddings permanently, allowing users to build a permanent library of their textbooks without re-embedding them every session.

Security & Environment Variables:
The backend heavily relies on purely server-side resolution of 'process.env.GEMINI_API_KEY'. By keeping all LLM interactions in the Next.js API routes, the client bundle remains completely ignorant of API secrets. We also added fallback behaviors (mock JSON responses) if the environment variables are accidentally missing, ensuring the application fails gracefully for local testing environments.

4. TECHNICAL STUFF HAPPENING IN THE BACKGROUND
----------------------------------------------
The underlying mechanisms that drive this application are heavily optimized and highly technical. Here is a deep dive into the hidden complexities of the application:

I) Cosine Similarity Vector Math
When searching for relevant textbook notes, the backend executes a manual cosine similarity function against a potentially huge array of 3072-dimensional floating-point vectors.
The formula implemented is: (A · B) / (||A|| * ||B||).
We iterate through every cached passage. We calculate the dot product of the user's message vector (A) and the textbook chunk vector (B). We then divide this by the product of their magnitudes. The resulting number between -1 and 1 represents conceptual semantic similarity. We rank the results, pop the top 3, and dynamically alter the system prompt of the AI to include these chunks. All of this happens in less than 300 milliseconds.

II) LLM Schema Enforcement
One of the massive technical hurdles in AI development is ensuring the language model returns usable data, not just pretty text. In '/api/chat', we enforce a strict output architecture using Google GenAI's 'responseSchema' configuration. We force the AI to return exactly:
{
  "reply": "string",
  "score": "integer"
}
However, because LLMs are non-deterministic, they occasionally wrap the JSON in markdown code blocks like "```json". We have background regex sanitizers and null-byte strippers operating invisibly before running 'JSON.parse()'. If parsing fails, the system catches the exception and falls back to a safe default message rather than crashing the chat UI.

III) Asynchronous Pipeline Management for Audio
The Xenova/Transformers.js integration for local Whisper transcription is incredibly heavy. When the Node server spins up, the model isn't loaded. The first time a user sends audio, the API route dynamically downloads the model weights from the Hugging Face hub (if not cached locally), compiles the computation graph, and holds the 'transcriber' pipeline in a global variable. Subsequent requests skip the loading phase and immediately execute raw Float32 array inference.

IV) Visual and UI Background Processing
The sleek interfaces use complex CSS algorithms. The 'ReactiveBackground.tsx' file is rendering a WebGL or Canvas-based animated gradient background that shifts dynamically based on user interaction or time.
The CSS Modules ('page.module.css', 'WikipediaModal.module.css') make extreme use of CSS Custom Properties (variables), rgba() alpha channels for glassmorphism panels, backdrop-filters for blurring elements behind modals, and intricate keyframe animations for hover states.

V) Next.js Client-Side Routing and Hydration
Because the application is heavily state-dependent, we use 'use client' directives at the top of our page components. However, this means we must carefully manage React Hydration. In 'page.tsx', we check 'if (!isMounted) return null;' before rendering to prevent server-side mismatch warnings when reading from 'localStorage' (which is completely unavailable on the server during the Next.js pre-rendering pass).

VI) The Game Loop Evaluator
In the chat interface, there is a hidden background evaluator tracking the "Mastery Score". As the user explains the concept, the AI's internal state machine updates the integer score from 0 to 100. This score drives UI progress bars and triggers achievements locally. If the score hits 100, the background loop flags the session as "mastered" and recalculates a spaced-repetition interval (SuperMemo-2 algorithm adaptation seen in the session object schema) for when the user should review the topic next.
